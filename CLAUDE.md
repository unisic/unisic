# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Build & Run

```sh
cmake -B build -G Ninja -DCMAKE_BUILD_TYPE=Release
cmake --build build
./build/unisic
```

Requires `qt6-qtbase-devel qt6-qtdeclarative-devel qt6-qtsvg-devel pipewire-devel` (Fedora) plus runtime `ffmpeg` and `wl-clipboard`. `pipewire-devel` is optional — without it the build succeeds but recording is disabled (`HAVE_PIPEWIRE` guard). Same pattern for OCR: `tesseract-devel leptonica-devel` + a langpack like `tesseract-langpack-pol` enable it (`HAVE_TESSERACT`); `zxing-cpp-devel` additionally enables QR/barcode decoding inside the OCR path (`HAVE_ZXING` — a code in the region copies its payload instead of OCR-ing its pixels). The capture popup positions itself by filling the screen and masking input to the card (no layer-shell dependency). `ctest --test-dir build` runs the QtTest targets in `tests/` (pure-logic units — version compare, shortcut format, annotation canvas, history filter); everything compositor-bound is covered by the in-app smoke test instead.

## Architecture

- `src/AppContext.*` — facade exposed to QML as context property `App`; owns all subsystems and the after-capture pipeline (editor/save/clipboard/upload/history), tray icon, hotkey dispatch.
- `src/capture/` — backends: `KWinScreenShot2` (silent, needs the installed .desktop file's `X-KDE-DBUS-Restricted-Interfaces`; auto-falls back on auth failure) and `PortalScreenshot`; `PortalRequest` implements the portal Request/Response handle pattern; `ScreenCastSession` drives the ScreenCast portal for recording; `CaptureManager` picks the backend.
- `src/editor/AnnotationCanvas` — the core drawing surface (QQuickPaintedItem): all tools, selection, undo/redo, and final compositing in **image-pixel space** (DPR forced to 1.0). Used by both the overlay and the editor; export is exactly what's rendered.
- `src/overlay/OverlayController` — freezes each screen (one capture per monitor), spawns one fullscreen `OverlayWindow.qml` per screen, returns annotated crop (screenshots) or physical-pixel region (GIF). **Smart pick and U-2-Net background removal were REMOVED in 0.7.1b** — do not reintroduce without asking. `src/overlay/ObjectDetector.*` (pure-pixel line-pair/flood-fill/enclosed-interior detection driving a click-to-pick region overlay), `src/segment/U2NetSegmenter.*`, the whole `HAVE_ONNX`/onnxruntime CMake path, the editor's "Remove background" action and the Settings model-download UI are all gone: the detector never matched real windows reliably enough to ship on (it was EXPERIMENTAL, default OFF), and no packaging channel ever built onnxruntime in, so `HAVE_ONNX` was OFF in every shipped package — U-2-Net only ever ran in local dev builds. `AnnotationCanvas::Tool` keeps `ObjectPick = 13` as a RESERVED hole so the persisted enum values of the tools after it do not shift. The editor's smart eraser (`SmartErase = 12`) is unrelated and stays: it rebuilds the background under the stroke from the four boundary strips just OUTSIDE the rect (never from inside it — the object is in there). Each strip sample is the per-channel median of a 5-along × 3-out window; then the strips are CLEANED — ring median + 3·1.4826·MAD band marks outliers, which are re-interpolated from the inliers bracketing them — and only then interpolated as a **Coons patch**, clamped per pixel to the range of the four samples it interpolates. Computed at ≤192² and rescaled (smooth by construction, so the rescale is free); reuses Blur's `fxPatch` cache + `m_fxFast` drag rule; samples via `pixelColor` (converting a 4K base per drag move would rewrite ~33 MB). **Every one of those steps is load-bearing, do not simplify them away**: up to 0.7 it filled with ONE averaged colour sampled from the rect's OWN border, which read as a smudge; a plain Coons over raw strips then smeared the object across the patch whenever a stroke edge cut through it (the normal case — scrubbing over text) and overshot into bright bands. `tests/AnnotationCanvasTest.cpp` pins all three: `smartEraseFollowsGradientBackground` (flat fill → drift 41/255), `smartEraseOverlappingStrokesStayOnBackground` (no outlier rejection → drift 210/255), `smartEraseRebuildsFlatBackground`.
- `src/record/` — `PipeWireGrabber` (libpipewire thread, keeps latest SHM frame) → `GifRecorder` samples at fixed FPS into ffmpeg stdin (lossless x264rgb temp), then two-pass palettegen/paletteuse GIF conversion. `RecordBorderHelper` — the GNOME record-region frame: mutter has no layer-shell and no keep-above for xdg_toplevel, so `unisic --record-border-helper` (dispatched in main() BEFORE QApplication — must boot on xcb, and one process can't host two QPA platforms) draws an XWayland **override-redirect** window (mutter's topmost stacking layer, click-through via `WindowTransparentForInput`). The region travels as monitor FRACTIONS: XWayland renames RandR outputs (`XWAYLAND<n>`) and its coordinate space (logical vs physical layout mode) need not match the Wayland one, but the fraction of the monitor is invariant; the helper pads 2 px outward so rounding can never bleed the frame into the ffmpeg crop. Helper lifetime = parent's stdin pipe (EOF → quit — a crashed parent can't orphan the frame). `UNISIC_RECORD_BORDER=helper` forces this path on KDE for testing; `capRecordBorder()` = layer-shell ∨ X11 session ∨ KWin ∨ `$DISPLAY`.
- `src/upload/UploadManager` — destinations from `~/.config/unisic/destinations.json` (.sxcu-like; `$text$`/`$json:path$`/`$regex:$` extraction); `type: "curl"` shells out to curl for FTP/SFTP. **Never compile an Imgur Client-ID in**: it identifies the *application*, so one shipped ID puts every user on one ~1250/day cap and makes whoever registered it answerable to Imgur for strangers' uploads. The builtin ships with no Authorization header; the user pastes their own ID (DestinationsPage swaps the raw headers-JSON field for a Client-ID field on any `api.imgur.com` destination), `startUpload` refuses an ID-less Imgur upload with an actionable message (Imgur itself answers a bare 429), and `ensureBuiltins` strips the placeholder ID that shipped up to 0.7 from stored configs.
- `src/history/HistoryStore` — the model itself stays a plain newest-first list; `src/history/HistoryFilterModel` (QSortFilterProxyModel, `QML_ELEMENT`) is the History page's search/filter layer (`searchText`, `kindFilter`, `favoritesOnly`, `uploadedOnly`). `kindFilter` matches `CategoryRole` (`"image"|"gif"|"video"|"replay"`), NOT `KindRole`: an instant-replay clip is an .mp4 that trims/plays/copies exactly like a recording, so it keeps kind `"video"` and separates only as a category. That needs `Entry::origin` (persisted `"origin": "replay"`), threaded from `GifRecorder::finished(path, fromInstantReplay)` — the export path is an .mp4 like any other, so nothing downstream can recover it. Entries written before 0.7.1 have no origin and fall back to the `Unisic_Replay_<stamp>.mp4` name every export gets. The page addresses entries by `EntryIdRole`, never by row (a proxy row means a different entry after a filter switch, and a batch delete shifts the rows behind it): `entryById`/`removeByIds`/`setFavoriteByIds`, plus `entryIds`/`entryIdsBetween`/`timestampAt` on the proxy for select-all, shift-range and the sticky date header (rows outside the viewport have no delegate, so QML cannot collect these itself).
- `src/notify/LayerShellNotifier` (layer-shell card) + `src/notify/NotificationHelper` (GNOME) — one `NotificationPopup.qml` and one `NotifCard.h` (style→size table, the settings the card reads, `effectiveSettings()` for the settings preview's unsaved overrides), two hosts. `notificationActionOrder` is a CSV of stable action ids; both hosts receive it in the same settings snapshot and QML normalizes it by dropping unknown/duplicate ids and appending newly introduced actions. Layer-shell → a card-sized overlay surface anchored to the corner. GNOME/mutter → the SAME QML on an XWayland **override-redirect** window in a second process (`--notification-helper`; one process can't host two QPA platforms), Steam-toast style. `AppContext::m_activePopupNotif` enforces one stylized card at a time; a newer capture dismisses the previous card through whichever host owns it, allowing an old helper process to exit normally. **The alternatives are dead ends, do not retry them**: a plain Wayland toplevel cannot place itself (no positioning protocol), and the fullscreen-surface-with-input-mask trick that works for the capture overlay renders **BLACK** on mutter — it unredirects fullscreen surfaces, so their transparency is never composited. Mutter can strand XWayland hover on "inside": once the pointer reaches a native Wayland surface, XWayland freezes its pointer position and even `XQueryPointer` keeps naming the card. Silence timeouts, pointer polling, and `org.gnome.Mutter.IdleMonitor` cross-checks are unreliable. Instead the helper keeps the existing 16 px transparent shadow gutter inside its X input shape, so crossing the visible card edge clears QML hover while the pointer still belongs to XWayland. Do not shrink the helper's input mask back to the exact card rect.
- `src/theme/ThemeController` — module QML singleton (`QML_SINGLETON`) owning the selected theme (persisted `ui/theme`) and bridging the live system palette (`qApp->palette()` + `QStyleHints::colorScheme()`) for the "system" theme. `ThemeController::instance()` shares the one engine-created object with `IconImageProvider`. **Do NOT `qmlRegisterSingletonInstance` into the `Unisic` URI** — that clobbers the module's other C++ QML_ELEMENT types (e.g. `AnnotationCanvas` becomes "not a type"). **Community themes**: `<config dir>/themes/*.json` (schema = Theme.qml's `_defs` seed: 8 required colors + `isDark`, every `_expand()` token an optional override; parser `src/theme/ThemeJson.h`, header-only, unit-tested). **Built-in split**: core `system`/`unisic`/`dark`/`light` are hardcoded in Theme.qml's `_defs` (`unisic` is the mandatory palette AND the fallback for every unresolvable id — do not remove); the decorative palettes (Catppuccin ×2, Dracula, Nord, Gruvbox) are shipped as REAL editable JSON files: `resources/themes/*.json` lives in qrc only as the SEED SOURCE, and `ThemeController::seedThemesFolder()` copies them into `<config>/themes/` ONCE (guarded by the `ui/themesSeeded` flag — deleting a seeded file keeps it gone, never re-created; existing files never clobbered). They then load through the normal custom-theme scan as `custom:<stem>` ids. `ThemeJsonTest::shippedThemesAllParse` parses every shipped JSON from the source tree so a broken seed fails tests, never a user. Resolution order in Theme.qml `pal`: core → `custom:` (incl. seeded decoratives) → unisic fallback. ThemeController scans + hot-reloads via QFileSystemWatcher (dir AND files re-armed each scan — the watcher drops deleted paths), exposes `customThemes`/`customDefs`/`customThemeErrors`, ids are `"custom:<file-stem>"`; Theme.qml resolves them in `pal` with a stock-palette fallback when the file is gone (selection kept). Editing the ACTIVE custom theme bumps `rev` so provider icons recolor. Recording overlays are theme tokens too (`recBadge{Bg,Text}`/`recDot`/`countdown{Bg,Number}`/`keystroke{Bg,Text}`, deliberately NOT palette-derived — they sit over recorded content): RecordBorder.qml reads them directly; the keystroke painter gets them via `GifRecorder::setKeystrokeThemeProvider` (lazy — the recorder outlives engine construction order); the GNOME helper gets them as 5 trailing HexArgb argv colors (alpha matters for pill/disc).
- `src/theme/IconImageProvider` — `image://icon/<name>?color=%23RRGGBB&sz=NN&v=<rev>`. System theme → `QIcon::fromTheme` (Breeze fallback theme set in main), else the bundled monochrome SVG in `resources/icons/sym/` recolored via `CompositionMode_SourceIn`. Pixmap provider ⇒ `UIcon` must stay `asynchronous: false` (runs on GUI thread; `QIcon::fromTheme`/`qApp->palette()` aren't thread-safe).
- `qml/` — fully custom SwiftUI-like design system (**no Kirigami/Breeze**; QQuickStyle forced to "Basic"): `Theme.qml` singleton (9 palettes computed from tokens; keep property names stable — 19 files depend on them) + `ToolCatalog.qml` singleton (single source of annotation tools **and their letter shortcuts**; both toolbars build from `visibleFor(ctx, hiddenTools)`, both keyboard hosts resolve through `toolForShortcut()`) + `components/` (UButton, USwitch, UIcon, `UComboBox` on a `Popup` parented to `Overlay.overlay` so dropdowns escape Flickable clipping, …). Icons are `iconName` (freedesktop names) via `UIcon`, not emoji. Windows are created from C++ via QQmlComponent with per-window QQmlContext (`editorSession`, `overlayController` context properties).

Recording: one `GifRecorder` parameterized by `Output{Gif,Mp4,WebM}` + `SourceType{Screen,Region,Window}`; Window source passes `types=WINDOW` to `ScreenCastSession` (portal window picker). GIF (region/screen) and video (screen/region/window → MP4/WebM) are separate sidebar pages/`AppContext` entry points. Recording overlays: `CursorOverlayPainter` (pointer/halo/ripples, only in CursorMetadata mode) and `KeystrokeOverlayPainter` (screenkey-style badge, `record/keystrokes`, works in EVERY cursor mode) both composite in `compositeCursorOverlay` into the reused scratch buffer — `m_lastFrame` stays clean (it is the sample-and-hold source; baking a badge in would freeze it on held frames). Key/click events come from `KeyCapture`/`ClickCapture` (libinput udev observers on their own threads, plain open no grab, `HAVE_LIBINPUT`, gated on `InputPermission::probe()` — needs the `input` group, degrades silently). Badge labels are PHYSICAL (US) key legends: evdev codes only, the compositor's xkb layout is not observable from a libinput observer. Global hotkeys: `GlobalHotkeys::defineAction` registers with KGlobalAccel flag `IsDefault(0x8)` only (preserves KCM edits; a separate `SetPresent(0x2)` call *clears* the binding on that daemon). **The IsDefault reply lies**: when the daemon has an active binding stored as "none" it echoes the requested keys while filling only the default column — the action stays dead. So `AppContext::defineHotkeys` runs `hotkeyBindStatus` on EVERY launch: real `shortcutKeys` queries; any action reported unbound while the app stores a key gets re-asserted via `setShortcut` (`SetPresent|NoAutoloading`, 0x6); a differing bound key is synced into the UI. A deliberate KCM unbind made while the app runs still sticks (the change signal empties the stored string). User edits/imports also push `setShortcut`. **Alternate hotkeys**: each action holds up to 4 bindings — stored as ONE comma-joined portable string ("Meta+Shift+S, Print"; QKeySequence chord limit), `keysFor` flattens the chords into the daemon's alternate-key list, `UShortcutList` (chips + "+ add key" recorder) edits it. The daemon REORDERS alternates in replies — every binding comparison must go through `GlobalHotkeys::sameBinding` (set-based), or reorders read as KCM edits/failures. **Non-KDE hotkeys** go through `PortalGlobalShortcuts` (org.freedesktop.portal.GlobalShortcuts): the portal pins app identity PER D-BUS CONNECTION at its first portal call, and on GNOME Qt's portal platform theme touches the Settings portal during QGuiApplication construction — the shared bus is already pinned to app id "" for terminal/AppImage launches, and gnome-control-center's shortcuts provider silently DISCARDS BindShortcuts from an empty app id (code 2 → hotkey UI collapses to the compositor-binds card). So ALL GlobalShortcuts traffic runs on a private bus connection whose first call is `Registry.Register` (re-issued after a portal restart — the association is in-memory portal-side). GNOME also enforces the spec's one-BindShortcuts-per-session; re-binds ride the fresh-session retry. `UNISIC_HOTKEY_BACKEND=portal` forces this path on a KDE box for testing.

Deliberate choices: keep working via portals on non-KDE; clipboard mirrored through `wl-copy` when present (Wayland reliability); recording region = full-monitor stream cropped in ffmpeg (portals can't select regions).

Clipboard history (KDE Plasma): `AppContext::copyImageToClipboard` — on Plasma/KWin (`org.kde.KWin` on the bus) images are re-asserted through **`KSystemClipboard`** (KF6::GuiAddons, optional `HAVE_KGUIADDONS`) with an empty **`x-kde-force-image-copy`** marker MIME attached (`makeForceImageMime`). Without that hint Klipper leaves the capture pasteable-now but never copies it into clipboard **history**, so the shot vanishes as soon as anything else is copied (issue #51). `QClipboard::setImage`/`wl-copy` advertise only `image/png` and cannot attach an empty marker; `KSystemClipboard` sets the whole `QMimeData` over the wlr data-control protocol, which also lands **without a focused window** — so the KDE branch skips the `wl-copy` mirror entirely (it bumps `m_clipboardSeq` so a stale deferred mirror can't clobber it). Spectacle/Flameshot do exactly this. Verified empirically: with the hint the image persists in the selection after the setter process exits (Klipper now owns/serves it, adding its own `application/x-kde-onlyReplaceEmpty`); without it, the selection reverts to the previous entry. The plain `QClipboard::setImage` still runs first so synchronous local reads (X11, the smoke test) keep working. Fix guarded by `clipboardHistoryHintCheck()` in the F8 smoke test + a **Klipper clipboard history** Developer button.

After-capture pipeline: every enabled action (copy/save/upload/editor) fires independently and immediately in `AppContext::finishCapture` — the editor never blocks the others. Two extra post-capture affordances live here too: (a) **copy last capture** — a regular always-registered hotkey (`copy-last`, default Meta+Shift+C, `hotkeys/copyLast`): `finishCapture` retains the newest screenshot PNG-encoded off-thread (`m_lastCaptureData`, megabytes not a pinned QImage) and `copyLastCapture()` decodes + copies on demand. This REPLACED the 2s global Ctrl+C grab (it stole ordinary copies right after a capture); startup still releases AND unregisters the legacy `quick-copy` action so upgraded installs can't keep a stale Ctrl+C grab or a phantom KCM row. Spectacle-parity in-window Ctrl+C: the region overlay (`OverlayController::confirmAndCopy` + `takeCopyRequested`, screenshot flow only) confirms the selection and forces a clipboard copy even with auto-copy off, and the editor's Ctrl+C copies the composited image. (b) **floating preview** — clicking a capture-notification thumbnail calls `CaptureNotification::preview()` → `AppContext::openPreview` which dumps the full image to a temp PNG and opens `PreviewWindow.qml` with a `PreviewController`. With layer-shell the surface is FULLSCREEN with the wl input region masked to the visible card (`setInputRect`, same pattern as the capture popup) so the card drags as a plain QML item — the only smooth option, layer surfaces can't be system-moved and per-event margin repositioning diverges; pin toggles overlay/top layer. Fallback: normal frameless window + `startSystemMove` + stays-on-top hint. Opacity slider fades the card item (NOT `Window.opacity` — qtwayland silently ignores it). Filenames come from `AppContext::makeFileName()` (template tokens `%date% %time% %datetime% %unix% %rand%`, format/quality from settings). Settings export/import (`AppContext::exportSettings/importSettings`) serializes all `Settings` Q_PROPERTYs via the metaobject (not raw QSettings keys — those omit defaults) plus destinations. Settings keys: NEVER put a key in a QSettings group named "general"/"General" — that name collides with INI's magic General section (serializes `[%General]`, parses back as `"General/..."`, case-sensitive reads miss → resets to defaults every launch). General-tab settings are top-level bare keys (plain `[General]` section) for exactly this reason. CLI: `unisic --fullscreen|--region|--window|--measure|--monitor|--recapture|--gif|--export-settings <f>|--import-settings <f>`. `--monitor` = screen under the cursor (`CaptureManager::captureActiveScreen`: KWin `CaptureActiveScreen` resolves the pointer's screen compositor-side; elsewhere the `QCursor::pos()` hint + portal workspace crop). `--recapture` = repeat the last region shot: the overlay stores each confirmed selection as `capture/lastRegion` (`"<screen>|<x>,<y>,<w>,<h>"` in LOGICAL px of the named screen — survives the KWin-native vs portal-scaled freeze difference), `recaptureLastRegion()` re-captures that screen and crops after rescaling to the current image size.

Developer mode: a `UNISIC_DEV_BUILD` CMake flag (auto ON for local builds — i.e. `UNISIC_BUILD == "dev"` — OFF for CI releases) exposes a **Developer tab** in Settings (`App.devBuild`). A dev build is also a **fully separate app** from installed stable: app name `unisic-dev` (which moves all QStandardPaths: cache, history, U-2-Net model), desktop id `app.unisic.UnisicDev`, own single-instance socket, own config dir `~/.config/unisic-dev` (seeded from stable on first run with all hotkeys UNBOUND — the stable KGlobalAccel component still owns its keys) and own hotkey component `unisic-dev`; the tray icon gets an amber corner dot. `ensureDesktopFile()` writes the dev entry and removes the legacy dev-written shadow of `app.unisic.Unisic.desktop` in `~/.local/share/applications` (which used to hijack the stable menu entry). Both apps can run and be CLI-targeted side by side. It shows detected compositor capabilities (`App.capNativeNotification/capCustomNotification/capRecordBorder`), a full smoke test bound to **F8** (`AppContext::runSmokeTest`), and a **per-action button** for each path. **REQUIREMENT for every new user-facing feature/path: wire it into BOTH** (a) the sequential smoke test in `AppContext::runSmokeTest` (as a pass/fail/skip step) **and** (b) a dedicated per-action button in the SettingsPage Developer pane (reuse the public `Q_INVOKABLE`, or add a small `devTest*()` for non-capture paths). This keeps the dev harness an exhaustive, hand-checkable inventory of what the app can do. In a release build, capability-gated options are shown greyed-out with an explanation rather than hidden.

## Internationalization (REQUIREMENT)

Every user-facing string MUST be wrapped in `qsTr()` (QML) / `tr()` (C++) **and** translated in all four `i18n/unisic_{en,pl,es,it}.ts` files. English is the source language — its `.ts` mirrors the source text (translation == source). Workflow for any new/changed string: (1) add the `qsTr`/`tr` call; (2) run `cmake --build build --target update_translations` — lupdate rescans C++ + `qml/*.qml` and appends the new strings as `type="unfinished"` with an empty `<translation>`; (3) fill every unfinished `<translation>` in **all four** files and drop the `unfinished` marker. A plain `cmake --build build` runs lrelease and bakes the `.qm` into the qrc (`HAVE_TRANSLATIONS`), so any `unfinished`/empty entry silently falls back to English at runtime — do not ship those. Keep placeholders (`%1`), filename tokens (`%date%`), file globs (`*.png`), and program names (`pw-play`/`paplay`/`aplay`) verbatim; match the per-language sound-cue convention already in the files (Shutter → Migawka / Obturador / Otturatore, Beep → Bip / Pitido / Bip, Off → Wyłącz / Desactivado / Spento, …). New QML component files must be added to `qt_add_qml_module` in `CMakeLists.txt` so lupdate scans them.

## What Unisic Is

Unisic is a planned screenshot and screen-recording tool for **Linux Wayland**, prioritizing KDE Plasma/KWin but portable via xdg-desktop-portal. Tech stack: **C++17+ with Qt 6, Qt Quick/QML UI**.

Required features (in rough build-priority order):

1. **Capture**: full screen (multi-monitor), interactive region (live selection with dimension readout), specific window.
2. **On-overlay annotation** during region selection (arrows, shapes, text) *before* the capture is finalized.
3. **Post-capture editor** auto-opened after every capture: arrows, lines, rectangles, ellipses, freehand, text, blur/pixelate, crop, numbered step markers, highlight, undo/redo.
4. **Modular upload destinations** (custom HTTP/FTP/SFTP/API + public services), auto-copy URL to clipboard, upload history.
5. **GIF screen recording** (distinct from normal video recording): record region/full screen, convert to .gif via ffmpeg; options for FPS, area, max duration.

Lower priority: tray icon + quick menu, configurable global hotkeys per capture mode, capture history with thumbnails.

## Hard Constraints

- **Wayland-legit capture paths only**: xdg-desktop-portal Screenshot/ScreenCast, KWin `org.kde.KWin.ScreenShot2` DBus API as a KDE-specific enhancement, PipeWire as the video backend, KGlobalAccel over DBus for global shortcuts, `QDBusInterface`/`QDBusConnection` for DBus. No X11-only capture hacks.
- **Mandatory UI palette**: Primary `#17153B` (main window/panel backgrounds), Secondary `#2E236C` and Tertiary `#433D8B` (secondary elements, hover/active), Accent `#C8ACD6` (action buttons, attention).

## Planning Workflow

`.claude/workflows/unisic-plan-research.js` (invocable as the `unisic-plan-research` workflow/skill) runs a three-phase multi-agent plan: 7 parallel research agents (portals, KWin/hotkeys, PipeWire, overlay selection, capture-tool parity, upload stack, GIF encoding) → 2 independent architecture designs (pragmatic vs KDE-native lens) → a synthesis into one final architecture + milestone plan. Its `REQ` constant is the canonical requirements statement for the project.
